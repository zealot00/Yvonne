# Yvonne KMS v1.3 合规功能指南

> 版本：v1.3.0 | 日期：2026-06-30

本文档覆盖 v1.3.0 新增的合规深化功能：MFA、Quorum、国密 TLS、OpenTelemetry、Config Reload、Alerting。

## 一、MFA TOTP（RFC 6238）

### 1.1 配置

```json
{
  "mfa": {
    "enabled": true,
    "issuer": "Yvonne KMS",
    "window_seconds": 30,
    "sensitive_operations": ["ShredKey", "EmergencySeal", "ExportKey", "SoftDeleteKey"]
  }
}
```

### 1.2 API

#### 注册 TOTP

```
POST /api/v1/auth/mfa/setup
Authorization: Bearer <token>

{"role_id": "admin"}
```

响应：

```json
{
  "ok": true,
  "data": {
    "secret": "JBSWY3DPEHPK3PXP",
    "uri": "otpauth://totp/Yvonne%20KMS:admin?secret=JBSWY3DPEHPK3PXP&issuer=Yvonne%20KMS&algorithm=SHA1&digits=6&period=30",
    "issuer": "Yvonne KMS",
    "account": "admin"
  }
}
```

用 QR code 库将 `uri` 生成二维码，用 Google Authenticator 扫描。

#### 验证 + 启用 MFA

```
POST /api/v1/auth/mfa/verify
Authorization: Bearer <token>

{"role_id": "admin", "code": "123456"}
```

验证通过后 MFA 启用，后续敏感操作需提供 `X-MFA-Code` header。

#### 禁用 MFA

```
POST /api/v1/auth/mfa/disable
Authorization: Bearer <token>
X-MFA-Code: 123456

{"role_id": "admin", "code": "123456"}
```

### 1.3 敏感操作 MFA 校验

启用 MFA 后，敏感操作需在请求头携带 `X-MFA-Code`：

```
DELETE /api/v1/keys/order-key/shred
Authorization: Bearer <token>
X-MFA-Code: 123456

{"version": 1}
```

无 `X-MFA-Code` 或 code 错误返回 `403 Forbidden`。

### 1.4 Policy 配置

在 `auth.app_roles` 中为角色启用 MFA：

```json
{
  "auth": {
    "app_roles": [
      {
        "role_id": "admin",
        "token": "admin-token",
        "policy": {
          "allowed_keys": ["*"],
          "allowed_actions": ["*"],
          "require_mfa": true
        }
      }
    ]
  }
}
```

## 二、Quorum Approval（K-of-N 审批）

### 2.1 场景

敏感操作（如 ShredKey）需 K 个审批者批准后才能执行。防单人误操作/恶意操作。

### 2.2 API

#### 创建审批 ticket

```
POST /api/v1/approvals
Authorization: Bearer <token>

{
  "operation": "ShredKey",
  "key_id": "order-key",
  "required": 2,
  "ttl_hours": 24
}
```

响应：

```json
{
  "ok": true,
  "data": {
    "id": "a1b2c3d4-...",
    "operation": "ShredKey",
    "status": "pending",
    "required": 2,
    "approvers": [],
    "expires_at": "2026-07-01T12:00:00Z"
  }
}
```

#### 审批通过

```
POST /api/v1/approvals/approve
Authorization: Bearer <approver-token>

{"ticket_id": "a1b2c3d4-..."}
```

达到 K 票后 `status` 变为 `approved`。

#### 审批拒绝

```
POST /api/v1/approvals/reject
Authorization: Bearer <approver-token>

{"ticket_id": "a1b2c3d4-..."}
```

任一拒绝者 reject 后 `status` 立即变为 `rejected`。

#### 查询 ticket

```
GET /api/v1/approvals?id=a1b2c3d4-...
Authorization: Bearer <token>
```

#### 列出 pending

```
GET /api/v1/approvals
Authorization: Bearer <token>
```

### 2.3 执行审批后的操作

审批通过后，携带 `X-Approval-Ticket-ID` 执行敏感操作：

```
DELETE /api/v1/keys/order-key/shred
Authorization: Bearer <token>
X-Approval-Ticket-ID: a1b2c3d4-...

{"version": 1}
```

### 2.4 安全规则

- **防自批准**：请求者不能审批自己的 ticket
- **审批幂等**：同一审批者重复 approve 返回 `409 Conflict`
- **过期清理**：超过 TTL 的 ticket 自动标记 `expired`
- **状态机**：`pending → approved / rejected / expired`，已完成拒绝再次操作

### 2.5 Policy 配置

```json
{
  "policy": {
    "require_quorum": 2,
    "approver_roles": ["approver-1", "approver-2", "approver-3"]
  }
}
```

## 三、RFC 8998 国密 TLS（GB/T 38636）

### 3.1 编译

国密 TLS 需 gmsm 构建标签：

```bash
go build -tags gmsm -o yvonne ./cmd/yvonne/
```

### 3.2 配置

```json
{
  "server": {
    "tls": {
      "enabled": true,
      "gm_enabled": true,
      "gm_sign_cert_file": "/path/to/sm2-sign.pem",
      "gm_sign_key_file": "/path/to/sm2-sign-key.pem",
      "gm_enc_cert_file": "/path/to/sm2-enc.pem",
      "gm_enc_key_file": "/path/to/sm2-enc-key.pem"
    }
  }
}
```

### 3.3 密码套件

- `GMTLS_SM2_WITH_SM4_SM3` — SM2 静态密钥交换 + SM4 + SM3
- `GMTLS_ECDHE_SM2_WITH_SM4_SM3` — ECDHE-SM2 + SM4 + SM3

### 3.4 非 gmsm 构建

非 gmsm 构建下 `gm_enabled=true` 会返回错误，需用标准 TLS。

## 四、OpenTelemetry tracing

### 4.1 配置

```json
{
  "observability": {
    "tracing": {
      "enabled": true,
      "endpoint": "localhost:4317",
      "service_name": "yvonne-kms"
    }
  }
}
```

### 4.2 架构

- OTLP gRPC exporter 导出 span 到 Jaeger/Tempo/OTel Collector
- `otelhttp` 自动为所有 HTTP 请求创建 span
- TraceID 从 OTel span context 提取，写入 `audit.LogEntry.TraceID`
- 支持分布式链路追踪（W3C TraceContext 传播）

### 4.3 禁用

`tracing.enabled=false` 时使用 noop tracer，无性能开销。

## 五、Config Reload（SIGHUP）

### 5.1 热更新字段

| 字段 | 可热更 | 说明 |
|---|---|---|
| `logging` | ✅ | 日志级别 |
| `audit` | ✅ | 留存天数、syslog |
| `crypto` | ✅ | 密码套件 |
| `mfa` | ✅ | MFA 配置 |
| `observability` | ✅ | tracing、alerting |
| `server` | ❌ | 端口、TLS（需重启） |
| `storage` | ❌ | 数据库连接（需重启） |
| `unseal` | ❌ | 解封配置（需重启） |
| `auth` | ❌ | 认证器（需重启） |

### 5.2 触发

```bash
kill -HUP <yvonne-pid>
```

日志输出：

```
SIGHUP received, reloading config...
config reloaded: hot fields updated (logging/audit/observability)
```

## 六、Alerting Webhook

### 6.1 配置

```json
{
  "observability": {
    "alerting": {
      "enabled": true,
      "webhook_url": "https://hooks.slack.com/services/...",
      "high_risk_operations": ["ShredKey", "EmergencySeal", "QuorumReject"]
    }
  }
}
```

### 6.2 支持的 Webhook

| 平台 | URL 检测 | 消息格式 |
|---|---|---|
| Slack | `hooks.slack.com` | `text` 字段 |
| 钉钉 | `oapi.dingtalk.com` | `markdown` 字段 |
| PagerDuty | `pagerduty.com` | Events API v2 |
| 通用 | 其他 | JSON |

### 6.3 告警事件

```json
{
  "alert": "yvonne-kms",
  "operation": "ShredKey",
  "actor": "admin",
  "resource": "order-key",
  "timestamp": "2026-06-30T12:00:00Z",
  "trace_id": "abc123",
  "description": "high-risk operation"
}
```
